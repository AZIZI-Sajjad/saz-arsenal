## Job Story
```
Fonctionnalité ne concernant par un utilisateur mais un système (informatique par exemple), comme par exemple l’’archivage automatique de données quotidiennement.
```

## Compliqué
```
Projet dont la demande initiale ne change pas (ou très peu) et dont la maîtrise technique est assurée. Il se réalise avec des étapes qui s'enchaînent les unes après les autres et peut se déconstruire. Ex .: assembler une voiture est un procédé industriel compliqué.
```

## Rationalisme
```
Courant de pensée qui s’oppose à l’empirisme (en termes de gestion de projet). Le rationalisme se base sur le raisonnement comme étant la seule source déterminante de la connaissance.
```

## Sprint
```
Timebox durant laquelle l'équipe est concentrée sur la production d'un incrément livrable.
```

## Empirisme
```
Courant de pensée qui s’oppose au rationalisme (en termes de gestion de projet). L’empirisme se base sur l’expérience comme étant la seule source déterminante de la connaissance.
```

## Vélocité
```
Données utilisée par une équipe technique pour indiquer sa capacité à faire à l'échelle d'un sprint.
```

## Product Owner 
```
Rôle proposé par le framework Scrum. La responsabilité de celui qui le porte est de déterminer "Quoi" faire sur le produit, diffuser la vision et représenter les users.
```

## Paradigme
```
Un paradigme est une représentation du monde, ou encore une manière de «&nbsp;voir les choses&nbsp;», avec son propre raisonnement.
```

## Chaos
```
Au-delà de "compliqué" et "complexe", le "chaos" est la zone (de la matrice de Stacey) dans laquelle il n'y a pas de maîtrise du "quoi" et du "comment".
```

## Stacey (Matrice) 
```
Modèle permettant d'évaluer le niveau de complexité d'une situation ou d'un projet selon la maîtrise de "quoi" et du "comment"
```

## Equipe technique
```
Rôle proposé par le framework Scrum. La responsabilité de l'équipe est de déterminer "Comment" réaliser le produit. Elle porte les choix techniques et est auto-organisée.
```

## User Story
```
Expression de besoin Agile. Elle se veut être un prétexte à la conversation. Elle se construit sur 3 questionnements : Pour qui («&nbsp;En tant que… »), pour quoi («&nbsp;Je veux/peux…&nbsp;»), pourquoi («&nbsp;Afin de…&nbsp;»).
```

## Burn-down
```
Indicateur montrant la quantité de travail restante à réaliser.
```

## Burn-up
```
Indicateur montrant l'acquisition de valeur.
```

## Complexe
```
Projet dont la demande initiale est peu claire ou changeante et dont la maîtrise technique n'est pas assurée. Il se réalise avec une démarque empirique afin d'avancer par petits pas significatifs pour valider les exigences fonctionnelles et techniques.
```

## Complexité (User Story) 
```
La complexité se traduit par l’effort nécessaire par l’équipe technique pour réaliser une User Story. Plus cet effort est élevé, plus la complexité l’est.
```

## Scrum Master
```
Rôle proposé par le framework Scrum. La responsabilité de celui qui le porte est de faciliter les interactions au sein de l'équipe.
```

## Valeur (User Story)
```
Valeur apportée par la User Story pour l’utilisateur. On parle de valeur business (utilité, sens) et non monétaire.
```

## Rétrospective
```
Temps durant lequel l'équipe réfléchit à ce qu'il s'est bien passé et moins bien passé durant le sprint, afin de s'améliorer.
```

## Chaos Report
```
Rapport publié par le Standish Group dévoilant la part des projets réussis, mitigés, en échec. De plus, il détaille les raisons des échecs des projets.
```

## Scrum
```
Framework Agile de développement de produit IT
```

## Timebox
```
Période de temps maximale dédiée à une activité (réunion, atelier, présentation…) à la durée déterminée à l’avance. A la fin du temps imparti, l’activité doit être arrêtée, quel que soit son avancement.
```

## Systémique
```
Vu comme étant un système qui se concentre sur les interactions des individus, des services et des flux entre eux. La systémique s'oppose aux approches locales qui consistent à observer les individus directement.
```

## Kanban
```
Framework (ou philosophie) Agile totalement basé sur le Lean (et le Toyotisme, dont il est la source). Kanban met en avant la limitation du travail en cours, la gestion des flux, le «&nbsp;Just-In-Time&nbsp;»
```

## Daily Meeting
```
Timebox quotidienne au cours de laquelle chacun participant répond brièvement à 3 questions (ce qu’il a fait hier, ce qu’il fera aujourd’hui, ses points de blocage)
```

## Démonstration
```
Temps durant lequel l'équipe, notamment le PO présente l'incrément produit par l'équipe aux parties prenantes afin de récolter du feedback.
```

## Backlog
```
Outil de travail du Product Owner. Regroupe, dans le cadre du Product Backlog, l’ensemble des User Stories qui constitue le produit. Dans le cadre du Sprint Backlog, les User Stories sur lesquelles l’équipe technique s’est engagée.
```

## Projet
```
Un projet est une suite d’actions, chaînées à travers un processus ou une méthodologie. Cette suite d’actions part d’une demande initiale et conduit vers une réalisation concrète, le tout étant mesuré dans le temps.
```

## eXtrem Programming (XP)
```
Cadre de travail Agile basé sur des pratiques simples mais poussées à l’extrême. XP se combine très bien avec des frameworks comme Scrum ou Kanban.
```

## Epic Story
```
Ensemble de User Stories sur le même thème. Les Epic Stories sont un moyen de regrouper les User Stories par thématique.
```
